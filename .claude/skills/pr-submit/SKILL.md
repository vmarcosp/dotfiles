# PR Submit

Skill for submitting code review comments to a GitHub PR, based on the triage file generated by `/pr-review`.

## Invocation

```
/pr-submit <pr-url-or-number> [path-to-review-file]
```

- If the file path is not specified, look for `review-pr-{number}.md` in the current directory.
- If the file doesn't exist, inform the user and suggest running `/pr-review` first.

## Workflow

### Step 1: Read the review file

Read the `review-pr-{number}.md` file and parse the issues.

Each issue has:
- File and line
- Severity
- Comment
- Code suggestion (optional)

### Step 2: Confirm with user

Show a summary of the issues that will be submitted:

```
Issues to submit on PR #{number}:
1. [critical] file.ts:42: Title
2. [important] other.ts:10: Title
3. [suggestion] more.ts:5: Title

Submit all? (y/n) or list numbers to submit only some (e.g.: 1,3)
```

Wait for user confirmation. The user can:
- Confirm all (`y`)
- Select which to submit (e.g.: `1,3,5`)
- Cancel (`n`)

### Step 3: Submit comments

For each approved issue, post as an inline review comment on the PR using `gh api`:

```bash
gh api repos/{owner}/{repo}/pulls/{number}/reviews \
  --method POST \
  -f event="COMMENT" \
  -f body="" \
  --jq '.id'
```

For inline comments, use the pull request reviews API with comments:

```bash
gh api repos/{owner}/{repo}/pulls/{number}/comments \
  --method POST \
  -f body="{comment_body}" \
  -f commit_id="{latest_commit_sha}" \
  -f path="{file_path}" \
  -f line={line_number} \
  -f side="RIGHT"
```

#### Comment body formatting

The comment must be sent EXACTLY as written in the review file, direct, in the user's tone.

If there's a code suggestion, use the GitHub suggestion format:

````
{comment text}

```suggestion
{suggested code}
```
````

If there's no code suggestion, send just the comment text.

### Step 4: Report results

After submitting all comments, show:

```
Review submitted on PR #{number}:
- {N} comments posted
- {N} failures (if any)

Link: {pr_url}
```

## Error Handling

- If `gh` is not authenticated, inform the user.
- If a comment fails (e.g.: line doesn't exist in the diff), report which issue failed and continue with the rest.
- If the review file is not found, suggest `/pr-review` first.

## Important Notes

- Never submit comments without explicit user confirmation.
- Preserve the exact tone and style of the comments. Don't "improve" or formalize them.
- If the diff has changed since the review (new commits), warn the user that line numbers may have shifted.
